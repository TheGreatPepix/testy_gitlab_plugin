# testy-gitlab-plugin

Плагин для [TestY](https://gitlab-pub.yadro.com/testy/testy), который запускает автотесты в GitLab CI из интерфейса TestY.

Сам плагин не выполняет тесты и не записывает результаты. Он выбирает нужные тесты, запускает конвейер GitLab CI
и передает ему параметры. 
Дальше работает [`testy-pytest-adapter`](https://pypi.org/project/testy-pytest-adapter/): 
он ставится в репозиторий с автотестами, запускает `pytest` и отправляет результаты обратно в TestY.

## Зачем нужен

Обычный сценарий такой:

1. В TestY выбран проект и тест-план c синхронизированными автотестами.
2. Пользователь нажимает **Run autotests**.
3. Плагин запускает GitLab CI для репозитория с автотестами.
4. Автотесты выполняются в GitLab.
5. Адаптер записывает результаты в тот же план TestY.

Плагин также умеет:

- создавать временные токены TestY для переменной `TESTY_TOKEN` в GitLab CI;
- удалять старую историю автотестов по настройкам проекта;
- хранить список последних запусков.

## Что где находится

| Компонент | Где работает | За что отвечает |
|---|---|---|
| `testy-gitlab-plugin` | в TestY | настройки GitLab, запуск GitLab CI, токены, чистка истории |
| `testy-pytest-adapter` | в репозитории автотестов | запуск `pytest` и отправка результатов в TestY |

## Установка

Плагин устанавливается как обычный Python-пакет для TestY. При старте TestY сам находит его через механизм плагинов.

1. Добавь пакет в зависимости серверной части TestY, например в `backend/testy/requirements/prod.in`:

   ```text
   git+https://github.com/TheGreatPepix/testy-gitlab-plugin.git@<ref>
   ```

2. Пересобери серверную часть и примени миграции:

   ```bash
   python manage.py migrate
   ```

3. Задай настройку `TESTY_PUBLIC_URL` в `testy\backend\testy\root\settings\common.py`.
   ```
   TESTY_PUBLIC_URL=https://testy.example.com
   ```
   Это внешний адрес TestY, на который GitLab CI сможет отправлять результаты. Плагин передает его в конвейер как `TESTY_URL`.

После установки плагин появится на странице **Plugins**. Его страницы доступны по адресу `/plugins/gitlab-runner/`.

## Настройка

Настройки привязаны к выбранному проекту TestY. Проект передается в адресе страницы как `?project=<id>`.

В плагине есть три страницы:

- **Config** - настройки подключения к GitLab и правила чистки истории.
- **Tokens** - создание и отзыв временных токенов TestY.
- **Runs** - запуск автотестов и список последних запусков.

На странице **Config** нужно указать:

- адрес GitLab;
- числовой идентификатор проекта GitLab;
- ветку или тег, которые нужно запускать;
- токен запуска GitLab CI из раздела GitLab **Settings -> CI/CD -> Pipeline triggers**;
- включено ли подключение для проекта.

## Запуск автотестов

На странице **Runs** показываются только те планы, которые можно запустить. План считается готовым, если в нем есть тесты и у каждого теста есть атрибут `automation_id`.

При запуске плагин передает в GitLab CI переменные:

- `TESTY_ENABLED`;
- `TESTY_URL`;
- `TESTY_PROJECT_ID`;
- `TESTY_PLAN_ID`;
- `TESTY_PYTEST_TARGETS`.

`TESTY_TOKEN` плагин не передает. Его нужно заранее добавить в GitLab как защищенную или скрытую CI/CD-переменную репозитория с автотестами.

После попытки запуска создается запись `PipelineRun`:

- `triggered` - GitLab CI успешно запущен;
- `error` - запуск не удался.

Текущий статус GitLab CI плагин не опрашивает. За ходом выполнения нужно смотреть по ссылке на GitLab.

## Токены TestY

Страница **Tokens** создает временный токен TestY для текущего пользователя. Полный токен показывается один раз, сразу после создания.

Этот токен нужно добавить в репозиторий автотестов как переменную GitLab CI `TESTY_TOKEN`. Каждый пользователь видит и отзывает только свои токены.

## Чистка истории

На странице **Config** можно настроить удаление старых данных автотестов:

- результатов `TestResult` старше заданного числа дней;
- старых версий кейсов и шагов, если на них уже не ссылаются оставшиеся результаты.

Чистка затрагивает только автотесты, то есть кейсы с атрибутом `automation_id`. Ручные тесты не удаляются.

Значение `0` отключает соответствующее правило. Перед удалением можно посмотреть предварительный результат через **Preview**, а ручной запуск выполняется кнопкой **Run cleanup now**.

## Требования к репозиторию автотестов

В репозитории с автотестами должны быть:

- установленный `testy-pytest-adapter`;
- переменная GitLab CI `TESTY_TOKEN`;
- настройка GitLab CI, которая запускает `pytest --testy ...` при запуске конвейера через токен;
- обработка переменных `TESTY_PLAN_ID` и `TESTY_PYTEST_TARGETS`, если они используются в команде запуска.

## HTTP API

| Метод | Путь | Что делает |
|---|---|---|
| `POST` | `plugins/gitlab-runner/api/run/` | проверяет выбранные тесты и планы, запускает GitLab CI |
| `GET` | `plugins/gitlab-runner/api/plan/<id>/run-status/` | показывает, можно ли запускать план |

### Запуск из фронтенда TestY

HTTP API можно вызывать напрямую, но для запуска автотестов из интерфейса TestY нужно добавить frontend-кнопки. Для этого примените патч из этого репозитория к рабочей копии TestY:

```bash
cd /path/to/testy
git apply /path/to/testy_gitlab_runner/patches/testy-frontend-run-autotests.patch
```

Патч добавляет кнопку запуска в заголовок тест-плана и пункт запуска для выбранных тестов в bulk actions. Он рассчитан на frontend TestY из `https://gitlab-pub.yadro.com/testy/testy/`.

### `POST plugins/gitlab-runner/api/run/`

Запускает автотесты для выбранных тестов или вложенных планов.

Запрос должен быть отправлен авторизованным пользователем. Тело запроса - JSON:

```json
{
  "plan": 1234,
  "tests": [
    56789
  ],
  "plans": []
}
```

Поля запроса:

| Поле | Тип | Обязательное | Что означает |
|---|---|---|---|
| `plan` | число | да | идентификатор основного тест-плана TestY. По нему определяется проект и подключение к GitLab |
| `tests` | список чисел | нет | идентификаторы отдельных тестов, которые нужно запустить |
| `plans` | список чисел | нет | идентификаторы тест-планов, из которых нужно взять тесты. Для каждого выбранного плана берутся и его вложенные планы |

`tests` и `plans` можно передавать вместе. Хотя бы один из этих списков должен быть непустым.

Плагин проверяет, что выбранные тесты и планы относятся к тому же проекту, что и `plan`. У каждого найденного теста должен быть заполнен атрибут `automation_id`: из него собирается список целей для `pytest`.

Успешный ответ возвращается со статусом `201`:

```json
{
  "id": 1,
  "gitlab_pipeline_id": 123456,
  "web_url": "https://gitlab.example.com/group/project/-/pipelines/123456",
  "status": "triggered",
  "targets": [
    "tests/test_example.py::test_name"
  ],
  "tests": [
    22128
  ],
  "plans": [],
  "cases": [
    1001
  ]
}
```

Если выбранные сущности не найдены или у тестов нет `automation_id`, API вернет `400` с описанием проблемы и списком проблемных идентификаторов. Если GitLab CI не удалось запустить, вернется `502`.

## Модели

- `GitlabConnection` - подключение TestY-проекта к GitLab.
- `PipelineRun` - один запуск GitLab CI из TestY.
- `HistoryRetentionPolicy` - правила чистки истории автотестов для проекта.
